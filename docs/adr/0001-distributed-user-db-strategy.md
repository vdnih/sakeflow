# ADR-0001: 分散型ユーザーDBと画像認識特化・エンティティ/イベント分離方針

- **ステータス**: 採用
- **決定日**: 2026-05-02
- **決定者**: vdnih

---

## 背景・課題

sakeflow-log はユーザーが飲んだお酒を記録するアプリである。記録機能を設計するにあたり、以下の3つのアーキテクチャ上の問題に対して方針を決める必要があった。

1. **銘柄データをどこで管理するか** — 全ユーザー共有の日本酒マスターDB（中央集権型）を構築するか、ユーザーごとに独立したデータとして保持するか。
2. **銘柄入力をどう補助するか** — テキストサジェスト（インクリメンタルサーチ）を提供するか、AI画像認識のみにするか。
3. **銘柄（エンティティ）と飲酒ログ（イベント）を同一コレクションで管理するか、分離するか。**

---

## 決定内容

以下3方針を採用する。

### 方針1: 分散型（per-user）DB戦略

中央集権的な日本酒マスターDBは構築しない。銘柄データはユーザーごとに独立したサブコレクション（`users/{userId}/sakes`）として保持する。

- 銘柄の表記揺れ（「作」「zaku」「ZAKU」など）はユーザー自身が修正する。
- 将来ソーシャル機能（ランキング・レコメンド）が必要になった場合は、ユーザーの承認を経て「パブリックミラーコレクション」へデータを昇格させるアーキテクチャを別途設計する。

**採用理由:**
- マスターデータの収集・更新・保守にかかる運用コストを完全に排除できる。
- テスト醸造酒・超限定品など未知の銘柄でもシステムエラーなく登録できる。
- Phase 1 において社会的機能は不要であり、オーバーエンジニアリングを避けられる。

**許容するトレードオフ:**
- システム全体でのデータ名寄せが困難なため「全体ランキング」「ユーザー間レコメンド」などのソーシャル機能の実装ハードルが上がる。
- 将来のDWH構築時、ETL処理でデータクレンジング負荷が増大する。

---

### 方針2: 画像認識特化・テキストサジェスト非採用

ユーザーの入力補完は **AIラベル画像認識のみ** に限定する。手入力フォームは提供するが、インクリメンタルサーチや候補サジェストUIは実装しない。

**採用理由:**
- 「ラベルを撮影する」という直感的な行動にユーザーを誘導でき、入力UXが統一される。
- 複雑な全文検索API・あいまい検索エンジンの開発・運用コストを削減できる。
- フロントエンド・バックエンドともにシンプルな構成を維持できる。

**許容するトレードオフ:**
- AI推論レイテンシ（数秒の待ち時間）が発生し、APIコスト（従量課金）が上昇する。
- 通信環境の悪い場所（地下の居酒屋、地方の酒蔵など）で機能が著しく制限される。オフライン対応は将来課題とする。

---

### 方針3: エンティティ（銘柄）とイベント（記録）の分離管理

「コレクション（銘柄の図鑑）」と「テイスティングノート（時系列の飲酒履歴）」を別のFirestoreコレクションとして分離して管理する。

| コレクション | パス | 役割 |
|---|---|---|
| sakes | `users/{userId}/sakes/{sakeId}` | 銘柄エンティティ（1銘柄1ドキュメント） |
| tasting_notes | `users/{userId}/tasting_notes/{noteId}` | 飲酒イベントログ（1回の飲酒1ドキュメント） |

- 銘柄の属性（特定名称・酒米・精米歩合・製法・フレーバー等）はカテゴリではなく **タグ配列** として保持する。
- tasting_notes 側には brand/brewery/prefecture/tags を **非正規化** して保持し、銘柄情報なしで読み込みを完結できるようにする（Read最適化）。
- 書き込み時（tasting_note 保存 ＋ sake 更新）は **Firestore WriteBatch** でアトミック性を確保する。

**採用理由:**
- 画面要件（コレクション画面と履歴画面）に応じた無駄のないデータ読み込みが可能になる。
- 銘柄とタグの分離により、複雑な日本酒属性を構造化しつつ表記揺れを吸収しやすくなる。

**許容するトレードオフ:**
- 「新しくお酒を飲んだ」という1アクションに対し、tasting_note 作成 ＋ sake 更新の2回書き込みが必要になる（WriteBatch で管理）。
- 非正規化によりデータ更新時に両コレクションを同期する必要があり、設計難易度が若干上がる。

---

## 結果として得られる設計

```
users/{userId}/
  sakes/{sakeId}
    brand, brewery, prefecture, category
    tasting_count, avg_rating
    first_drank_at, last_drank_at
    image_url, created_at, updated_at

  tasting_notes/{noteId}
    sake_id (→ sakes への参照)
    status: "processing" | "ready" | "failed"
    image_url, drank_at, job_id
    brand, brewery, prefecture, category, tags  ← 非正規化
    rating, note  ← ユーザー入力
    created_at, updated_at
```
